# V1.17 — исправление запуска компилятора из tools и прямой C++ сборки

Исправлено по логу, где `run_tests` видел `g++.exe`, `cmake.exe`, `ninja.exe` в `tools/windows/x64/msys64/mingw64/bin`, но запускал CMake абсолютным путём с кавычками. Через `cmd.exe` это превращалось в ошибку вида `"\"N:\\...\\cmake.exe\"" не является внутренней или внешней командой`.

## Что изменено

- На Windows команды для утилит из `tools` теперь используют короткое имя (`cmake.exe`, `g++.exe`, `clang++.exe`) вместо кавыченного абсолютного пути, потому что нужные папки уже добавляются в начало `PATH`.
- `run_tests` для C++ сначала пробует прямую сборку через `g++.exe`, `clang++.exe` или `cl.exe`, если они найдены в `tools` или PATH.
- CMake используется только если прямой C++ компилятор не найден.
- Прямая C++ сборка собирает все `.cpp/.cc/.cxx` файлы проекта, кроме `.cppagent` и `build`.
- Если CMake всё же нужен, `CMakeLists.txt` автоматически чинится под ожидаемый target `agent_app`, иначе запуск после сборки мог искать неправильный exe.
- В prompt добавлено правило: если найден `g++.exe/clang++.exe`, не считать ошибку CMake причиной остановки, а компилировать напрямую.

## Проверка

